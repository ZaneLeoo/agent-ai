# Dify 节点命名规范

更新时间：2026-07-01

## 目的

企业智能体前端会把 Dify 工作流节点展示为“处理步骤”。如果节点名称随意填写，用户会看到难以理解的原始节点名，前端也很难稳定判断节点含义。

本规范用于统一 Dify 工作流节点命名，让节点名称同时满足三个目标：

- 业务人员看得懂节点在做什么。
- 前端可以根据节点前缀稳定映射展示类型。
- 后端和前端在 Dify 工作流调整后仍能保持兼容。

## 命名格式

所有需要展示给用户的 Dify 节点，统一使用以下格式：

```text
[类型] 业务名称
```

示例：

```text
[输入] 用户输入
[意图] 问题分类
[图表] 图表生成
[模型] 普通问题回答
[知识] 销售知识检索
[模型] 知识库答案整理
[回答] 知识库结果回复
```

`[类型]` 是稳定标识，前端优先解析它。`业务名称` 是展示给用户和配置人员看的语义说明。

## 第一版支持类型

| 类型 | 用途 | 前端建议展示 |
| --- | --- | --- |
| `[输入]` | 工作流开始、接收用户输入 | 用户输入 |
| `[意图]` | 分类器、路由判断、问题类型识别 | 识别意图 |
| `[知识]` | 知识库检索、文档召回 | 检索知识库 |
| `[模型]` | 大模型回答、总结、分析、整理 | 大模型思考 |
| `[图表]` | 生成图表配置或图表数据 | 生成图表 |
| `[表格]` | 生成表格结构或明细数据 | 生成表格 |
| `[工具]` | 调用外部接口、数据库、HTTP、代码执行等工具 | 调用工具 |
| `[回答]` | 最终回复、直接回复、结果输出 | 整理回答 |

后续如工作流复杂度提升，可以扩展：

```text
[判断]
[汇总]
[校验]
[异常]
```

扩展类型需要同步更新前端映射规则和本文档。

## 当前工作流推荐命名

基于当前 Dify 工作流，推荐保持以下节点命名：

| 当前节点 | 推荐名称 | 说明 |
| --- | --- | --- |
| 开始节点 | `[输入] 用户输入` | 表示用户问题进入工作流 |
| 分类器/意图识别 | `[意图] 问题分类` | 用于判断走图表、普通回复或知识库分支 |
| 图表生成节点 | `[图表] 图表生成` | 生成图表数据或图表配置 |
| 图表直接回复 | `[回答] 图表结果回复` | 告知用户图表已生成 |
| 普通问题大模型 | `[模型] 普通问题回答` | 处理不需要知识库和图表的普通问答 |
| 普通问题直接回复 | `[回答] 普通结果回复` | 输出普通问答结果 |
| 知识库检索 | `[知识] 销售知识检索` | 从销售知识库召回来源 |
| 知识库答案大模型 | `[模型] 知识库答案整理` | 基于召回内容整理答案 |
| 知识库直接回复 | `[回答] 知识库结果回复` | 输出带知识库依据的最终答案 |

## 分类分支命名

分类器分支也应使用业务语义，不建议保留 `CLASS 1`、`CLASS 2`、`CLASS 3` 这类默认名称。

推荐分支名称：

```text
图表生成
普通回复
销售相关问题
```

如果 Dify 无法稳定修改分支名称，前端可以临时做兜底映射，但不应把 `CLASS n` 作为长期标准。

兜底示例：

| Dify 默认分支 | 临时映射 |
| --- | --- |
| `CLASS 1` | 图表生成 |
| `CLASS 2` | 普通回复 |
| `CLASS 3` | 知识库问答 |

## 前端解析规则

前端解析节点名称时按以下优先级处理：

1. 优先解析标准前缀，例如 `[知识] 销售知识检索` 识别为知识库节点。
2. 如果没有标准前缀，则使用关键词兜底，例如节点名包含“知识检索”时识别为知识库节点。
3. 如果仍无法识别，则展示 Dify 原始节点名，并按通用步骤处理。

推荐伪代码：

```ts
function resolveNodeType(title: string) {
  const prefix = title.match(/^[(\[](.+?)[)\]]/)?.[1]
  if (prefix) return prefix

  if (title.includes('知识')) return '知识'
  if (title.includes('图表')) return '图表'
  if (title.includes('表格')) return '表格'
  if (title.includes('意图') || title.includes('分类')) return '意图'
  if (title.includes('回复') || title.includes('回答')) return '回答'
  if (title.includes('模型')) return '模型'
  return '通用'
}
```

## 配置约束

- 节点类型前缀必须放在节点名称开头。
- 前缀使用中文方括号风格，例如 `[知识]`，不要混用 `知识-`、`知识:`、`【知识】`。
- 业务名称应说明节点真实职责，不要只写“大模型1”“直接回复2”。
- 同一个工作流内，相同职责的节点类型要保持一致。
- 最终直接输出给用户的节点统一使用 `[回答]`。
- 生成可视化数据的节点统一使用 `[图表]` 或 `[表格]`，不要用 `[模型]` 代替。
- 知识库召回节点统一使用 `[知识]`，后续前端会基于该类型展示来源和召回过程。

## 好与不好的示例

推荐：

```text
[意图] 问题分类
[知识] 销售知识检索
[模型] 知识库答案整理
[回答] 知识库结果回复
```

不推荐：

```text
大模型L1
直接回复 2
CLASS 3
知识检索
```

其中 `知识检索` 虽然能被关键词兜底识别，但缺少标准前缀，不利于长期维护。

## 与前后端的关系

这份规范不要求 Dify 返回额外字段。当前阶段前端可以仅根据节点名称解析展示类型。

后续如果后端能在 SSE `node` 事件中补充标准字段，推荐增加：

```json
{
  "nodeType": "knowledge",
  "displayName": "销售知识检索"
}
```

届时解析优先级调整为：

1. 优先使用后端标准字段。
2. 其次解析 Dify 节点名称前缀。
3. 最后使用关键词兜底。

这样可以进一步减少前端对节点名称文本的依赖。
